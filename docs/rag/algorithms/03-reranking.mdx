# 重排序

向量检索（Bi-Encoder）速度快但精度有限。它的本质是"分别把查询和文档编码成向量，然后比较向量的相似度"——这个过程丢失了查询和文档之间的细粒度交互信息。

**重排序（Reranking）** 的思路是：先用快速的向量检索召回候选集（如 Top-100），再用更精确的模型对这 100 个候选逐一打分排序，最后取 Top-K。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   查询        │────▶│  快速召回      │────▶│  精确重排序    │
│  "RAG原理"   │     │  Bi-Encoder   │     │  Cross-Encoder│
└──────────────┘     │  Top-100      │     │  Top-10       │
                     └──────────────┘     └──────────────┘
                        毫秒级              百毫秒级
                        高召回              高精度
```

## Bi-Encoder vs Cross-Encoder

| 特性 | Bi-Encoder | Cross-Encoder |
|------|-----------|---------------|
| **原理** | 查询和文档分别编码 | 查询和文档一起编码 |
| **速度** | 快（向量内积） | 慢（每次都要过模型） |
| **精度** | 中 | 高 |
| **适用环节** | 召回 | 精排 |
| **比喻** | 相亲前看简历照片 | 面对面交流 |

### 为什么 Cross-Encoder 更精确

```python
# Bi-Encoder: 查询和文档各自独立编码
query_embedding = embed("什么是 RAG?")
doc_embedding = embed("检索增强生成是一种...")
similarity = cosine_similarity(query_embedding, doc_embedding)
# 问题："RAG" 这个词在查询和文档中的语义可能被各自上下文稀释

# Cross-Encoder: 查询和文档一起输入模型
# 模型可以看到 "什么是" 和 "检索增强生成" 之间的 attention 关系
score = cross_encoder("什么是 RAG?", "检索增强生成是一种...")
# 模型能捕捉 "什么是" → "是一种..." 这种细粒度匹配
```

## Cross-Encoder 重排序

### 使用 sentence-transformers

```python
from sentence_transformers import CrossEncoder

# 加载预训练的 Cross-Encoder
reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')

# 候选文档（来自向量检索的 Top-100）
candidates = [
    "RAG 是 Retrieval-Augmented Generation 的缩写...",
    "大语言模型是一种深度学习模型...",
    "检索增强生成结合了信息检索和文本生成...",
]

query = "什么是 RAG?"

# 成对打分
pairs = [(query, doc) for doc in candidates]
scores = reranker.predict(pairs)

# 按分数排序
ranked = sorted(
    zip(candidates, scores),
    key=lambda x: x[1],
    reverse=True,
)

for doc, score in ranked[:5]:
    print(f"分数: {score:.4f} | {doc[:60]}...")
```

### 主流 Cross-Encoder 模型

| 模型 | 特点 | 适用场景 |
|------|------|----------|
| `cross-encoder/ms-marco-MiniLM-L-6-v2` | 轻量、英文 | 英文检索、资源受限 |
| `cross-encoder/ms-marco-electra-base` |  heavier、更精确 | 英文生产环境 |
| `BAAI/bge-reranker-base` | 中文优化 | 中文 RAG |
| `BAAI/bge-reranker-large` | 更大、更精确 | 中文生产环境 |
| `cohere/rerank-english-v3.0` | Cohere API | 云部署、不想本地跑模型 |

## ColBERT：Late Interaction

Cross-Encoder 精度高但太慢（每次都要过一遍大模型）。ColBERT 提出了一种折中方案：

```
Bi-Encoder:    Q → [向量]    D → [向量]    → 相似度 = 向量点积
Cross-Encoder: Q + D → [模型] → [分数]     → 精确但慢

ColBERT:       Q → [向量1, 向量2, ...]    ← 每个 token 一个向量
               D → [向量A, 向量B, ...]
               → 相似度 = MaxSim（细粒度匹配后聚合）
               → 比 Bi-Encoder 精确，比 Cross-Encoder 快
```

ColBERT 的核心是 **Late Interaction（延迟交互）**：查询和文档各自编码（像 Bi-Encoder 一样快），但在相似度计算时做细粒度交互（像 Cross-Encoder 一样精确）。

```python
# ColBERT 概念示意（实际需用官方实现）
from colbert import Searcher

# 索引
searcher = Searcher(index="my_index")

# 检索（内部使用 MaxSim）
results = searcher.search(query="什么是 RAG?", k=10)
```

## 两阶段检索流程

```python
class TwoStageRetriever:
    """两阶段检索：向量召回 + Cross-Encoder 重排序。"""

    def __init__(self, vector_store, reranker_model: str, initial_k: int = 100):
        self.vector_retriever = vector_store.as_retriever(
            search_kwargs={"k": initial_k}
        )
        self.reranker = CrossEncoder(reranker_model)
        self.initial_k = initial_k

    def retrieve(self, query: str, final_k: int = 5) -> list[tuple[Document, float]]:
        # Stage 1: 快速召回
        candidates = self.vector_retriever.invoke(query)
        print(f"Stage 1 召回: {len(candidates)} 个候选")

        if len(candidates) <= final_k:
            return [(doc, 1.0) for doc in candidates]

        # Stage 2: 精确重排序
        pairs = [(query, doc.page_content) for doc in candidates]
        scores = self.reranker.predict(pairs)

        # 组装结果
        ranked = sorted(
            zip(candidates, scores),
            key=lambda x: x[1],
            reverse=True,
        )

        return ranked[:final_k]

# 使用
retriever = TwoStageRetriever(
    vector_store=vectorstore,
    reranker_model="BAAI/bge-reranker-base",
    initial_k=50,
)

top_results = retriever.retrieve("什么是 RAG?", final_k=5)
for doc, score in top_results:
    print(f"重排序分数: {score:.4f} | {doc.page_content[:80]}...")
```

## 延迟与吞吐权衡

| 方案 | 延迟 | 相对精度 | 适用场景 |
|------|------|----------|----------|
| 仅向量检索 | ~10ms | 1.0x | 延迟极敏感 |
| 向量 + MiniLM 重排 | ~50ms | 1.15x | 通用推荐 |
| 向量 + Large 重排 | ~200ms | 1.25x | 精度优先 |
| 向量 + API 重排 | ~300ms | 1.30x | 不想运维模型 |
