# 什么是 RAG

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将外部知识检索与大语言模型（LLM）生成能力相结合的技术架构。它的核心思想很简单：在让模型回答问题之前，先从知识库中检索相关信息，然后把检索到的内容作为上下文喂给模型，让模型基于这些事实来生成回答。

## 为什么需要 RAG

大语言模型虽然知识广博，但存在几个根本性的限制：

| 问题 | 说明 |
|------|------|
| **知识截止** | 模型训练数据有截止日期，无法回答最新事件 |
| **幻觉** | 模型会自信地编造不存在的事实 |
| **领域盲区** | 对专业领域（如公司内部文档、私有数据）一无所知 |
| **无法溯源** | 无法说明答案来自哪里，难以验证 |

RAG 通过在生成前引入**检索步骤**，让模型能够接触到模型权重之外的信息，从而缓解上述问题。

## RAG 的基本架构

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   用户提问       │────▶│  检索相关文档    │────▶│  LLM 生成回答   │
│  "什么是RAG?"   │     │  (知识库/向量库) │     │  (基于检索上下文)│
└─────────────────┘     └─────────────────┘     └─────────────────┘
                                │
                                ▼
                        ┌─────────────────┐
                        │   外部数据源      │
                        │  文档/网页/数据库  │
                        └─────────────────┘
```

整个流程可以拆解为三个核心阶段：

1. **索引（Indexing）** — 将外部数据切分、向量化后存入向量数据库
2. **检索（Retrieval）** — 将用户查询向量化，在向量库中查找最相似的文档片段
3. **生成（Generation）** — 将检索到的文档片段与用户问题拼接，送入 LLM 生成回答

## RAG vs 微调

很多初学者会问：既然要让模型具备特定领域知识，为什么不直接微调模型？

| 维度 | RAG | 微调（Fine-tuning） |
|------|-----|---------------------|
| **数据更新** | 实时更新知识库即可 | 需要重新训练 |
| **成本** | 低（检索+推理） | 高（训练计算资源） |
| **幻觉控制** | 答案可溯源至原文 | 仍可能编造 |
| **适用场景** | 知识频繁变化、需要溯源 | 需要改变模型行为/风格 |
| **典型耗时** | 毫秒级 | 小时到天数 |

> 两者并非互斥。在实际工程中，RAG 负责注入最新知识，微调负责调整模型的输出风格和推理能力，组合使用效果最佳。

## 一个简单的 RAG 示例

假设我们有一个关于公司产品的知识库，用户问：

> "你们最新的产品支持哪些文件格式？"

没有 RAG 时，模型可能会一本正经地胡说。有了 RAG：

1. **检索**：向量库找到产品文档中关于"文件格式支持"的段落
2. **增强上下文**：
   ```
   [检索到的文档]
   我们的产品 v2.3 支持以下文件格式：PDF、DOCX、TXT、
   Markdown、CSV、JSON。v2.4 更新中新增了 XLSX 支持。

   [用户问题]
   你们最新的产品支持哪些文件格式？
   ```
3. **生成**：模型基于事实回答，并可以引用来源
