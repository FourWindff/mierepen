# 详细说明

一个完整的 RAG 系统通常由以下几个核心组件构成：

## 1. 文档加载与预处理

将各种格式的原始文档转换为可处理的文本块：

| 格式 | 处理方式 | 注意事项 |
|------|---------|---------|
| PDF | 提取文本 + OCR | 注意表格和图片 |
| Word | 解析 XML 结构 | 保留标题层级 |
| Markdown | 直接解析 | 最推荐的格式 |
| HTML | 去除标签 | 保留语义结构 |

<Callout type="warning" title="分块策略">
分块是 RAG 中最关键的决策之一。块太小会丢失上下文，块太大会增加噪声。常见的策略有：

- **固定长度**：每 512 tokens 一块
- **语义分块**：按段落或章节边界分块
- **重叠分块**：相邻块之间有 20% 的重叠内容
</Callout>

## 2. 嵌入模型

将文本转换为向量表示，以便进行相似度检索：

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('BAAI/bge-large-zh')
embeddings = model.encode(chunks)
```

## 3. 向量存储

选择合适的向量数据库来存储和检索向量：

<Callout type="info" title="选型建议">
- **Chroma**：轻量级，适合原型开发
- **Milvus**：高性能，适合大规模生产环境
- **PGVector**：PostgreSQL 扩展，适合已有 PG 基础设施的团队
</Callout>

## 4. 检索策略

检索是整个 RAG 流程的核心环节：

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| 向量检索 | 语义相似度匹配 | 概念性问题 |
| 关键词检索 | BM25 / TF-IDF | 精确术语查询 |
| 混合检索 | 向量 + 关键词融合 | 通用场景 |
| 重排序 | 先用向量召回，再用模型精排 | 高精度要求 |

<Collapse question="什么是重排序（Reranking）？">
重排序是一种两阶段检索策略：

1. **召回阶段**：使用轻量级的向量检索或关键词检索，快速召回 Top-K 候选文档
2. **精排阶段**：使用专门的交叉编码器（Cross-Encoder）模型，对召回的文档进行更精确的 relevance 打分，重新排序

这种方式兼顾了效率和精度，是目前生产环境中最常用的策略。
</Collapse>
