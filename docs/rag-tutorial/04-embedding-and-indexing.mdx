# 嵌入模型与向量索引

嵌入（Embedding）是 RAG 系统的核心纽带——它将文本转换为数值向量，使得语义相似性可以通过数学计算来衡量。向量索引则让这些向量的高效检索成为可能。本章深入讲解这两个关键技术。

## 嵌入模型基础

### 什么是嵌入

嵌入是一种将离散的文本（单词、句子、文档）映射到连续向量空间的技术。在这个空间中，语义相似的文本距离更近。

```
"猫"  →  [0.2, -0.5, 0.8, ...]  (768维)
"狗"  →  [0.3, -0.4, 0.7, ...]  ← 距离"猫"很近
"汽车" →  [-0.8, 0.2, -0.1, ...] ← 距离"猫"很远
```

关键特性：

- **语义保持**：相似语义的文本在向量空间中距离近。
- **可计算**：可以计算余弦相似度、欧氏距离等来量化相似性。
- **固定维度**：无论输入文本多长，输出都是固定维度的向量。

### 双编码器（Bi-Encoder）架构

RAG 中使用的嵌入模型绝大多数基于双编码器架构：查询和文档各自独立编码为向量，然后通过向量相似度匹配。

```
查询文本 ──→ [Encoder] ──→ 查询向量 ──┐
                                      ├──→ 相似度计算
文档文本 ──→ [Encoder] ──→ 文档向量 ──┘
```

**优点**：文档向量可以预先计算并存储，检索时只需编码查询，速度快。
**缺点**：编码时查询和文档没有交互，无法捕捉精细的匹配关系（这就是需要重排序的原因）。

## 嵌入模型选择

### 主流模型对比

| 模型 | 维度 | 最大长度 | 语言支持 | 特点 | 适用场景 |
|------|------|---------|---------|------|---------|
| text-embedding-3-small | 1536 | 8192 | 多语言 | OpenAI 出品，性价比高 | 通用场景，预算敏感 |
| text-embedding-3-large | 3072 | 8192 | 多语言 | 质量最好，维度高 | 追求最高精度 |
| BGE-M3 | 1024 | 8192 | 多语言（含中文） | 开源，支持稀疏向量 | 中文场景，混合检索 |
| E5-mistral-7b-instruct | 4096 | 32768 | 多语言 | 基于 Mistral，指令微调 | 长文档，复杂语义 |
| GTE-large | 1024 | 512 | 多语言 | 阿里出品，中文效果好 | 中文 RAG |
| Cohere Embed v3 | 1024 | 512 | 多语言 | 支持多种嵌入类型 | 多语言商业应用 |
| jina-embeddings-v3 | 1024 | 8192 | 多语言 | 支持分类、聚类等任务 | 多任务场景 |

### 选择建议

**维度**：

- 更高维度通常意味着更强的表达能力，但存储和计算成本也更高。
- 对于百万级以下的文档，1536 维（OpenAI small）通常足够。
- 对于千万级以上的大规模检索，可以考虑 768 或 1024 维的模型以节省成本。

**语言**：

- 如果知识库以中文为主，强烈建议使用针对中文优化的模型（如 BGE-M3、GTE）。
- 通用多语言模型（如 OpenAI 的 embedding 系列）在中文上表现尚可，但不如专门优化的模型。

**最大长度**：

- 嵌入模型有最大输入长度限制。如果分块超过限制，超出部分会被截断。
- 如果文档普遍较长，选择支持 8192 token 以上的模型。

### 自定义微调

如果领域非常特殊（如法律、医学、特定企业内部术语），可以考虑对嵌入模型进行领域微调。

**微调数据格式**：

```python
# 每个训练样本是一个 (查询, 正例, 负例) 三元组
train_data = [
    {
        "query": "如何申请专利？",
        "positive": "专利申请需要向国家知识产权局提交申请书、说明书和权利要求书...",
        "negative": "商标注册需要向商标局提交申请材料..."  # 不相关的文档
    },
    # ...
]
```

<Callout type="info" title="何时微调">
  只有当通用模型在你的领域上表现明显不佳，且你有足够的高质量标注数据（至少数千对查询-文档对）时，才考虑微调。对于大部分场景，选择一个优质的预训练模型就足够了。
</Callout>

### 嵌入 API 与本地部署

**云端 API**（如 OpenAI、Cohere、百度千帆）：

- 优点：无需维护，自动更新，按量计费。
- 缺点：数据需要发送到第三方，有隐私顾虑；有调用频率和并发限制；长期成本高。

**本地部署**（如使用 Hugging Face Transformers、Sentence-Transformers）：

- 优点：数据不出域，无调用限制，一次性硬件投入。
- 缺点：需要 GPU 资源，需要自行维护和升级。

```python
# 本地部署示例（使用 sentence-transformers）
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('BAAI/bge-m3')
embeddings = model.encode(["这是一个测试句子", "这是另一个句子"])

# 对于 BGE 模型，需要在查询前添加指令前缀
query_embedding = model.encode(
    ["为这个句子生成表示以用于检索相关文章：如何申请专利？"]
)
```

## 向量相似度度量

两个向量之间的相似度有多种计算方式：

### 余弦相似度（Cosine Similarity）

最常用。衡量两个向量方向的夹角，范围 [-1, 1]。

```python
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
```

- **优点**：不受向量长度影响，只关注方向。
- **适用场景**：文本语义匹配，大多数 RAG 系统的默认选择。

### 点积（Dot Product）

计算两个向量的点积。如果向量已经归一化，点积等于余弦相似度。

- **优点**：计算更快（少一次除法）。
- **适用场景**：向量已经归一化的情况，或需要保留向量长度信息时。

### 欧氏距离（Euclidean Distance）

衡量向量之间的直线距离。距离越小越相似。

- **适用场景**：空间聚类、需要绝对距离感的场景。

<Callout type="warning" title="度量一致性">
  索引构建和检索时必须使用相同的相似度度量。不同的度量在不同数据库中可能有不同的名称，配置时需仔细核对。
</Callout>

## 向量数据库

向量数据库是专门用于存储和检索高维向量的数据库系统。它们内置了高效的近似最近邻（ANN）搜索算法。

### 主流向量数据库对比

| 数据库 | 部署方式 | 特点 | 适用场景 |
|--------|---------|------|---------|
| **Pinecone** | 全托管 SaaS | 零运维，自动扩缩容，功能简洁 | 快速启动，不愿维护基础设施 |
| **Weaviate** | 托管 / 自托管 | 支持混合搜索（向量+BM25），GraphQL | 需要丰富查询能力 |
| **Milvus / Zilliz** | 自托管 / 托管 | 专为大规模向量设计，支持十亿级 | 超大规模，企业级 |
| **Qdrant** | 自托管 / 托管 | 轻量高效，Rust 编写，过滤能力强 | 中小型项目，过滤需求多 |
| **Chroma** | 嵌入式 / 服务器 | 极简 API，本地优先 | 原型开发，小型项目 |
| **pgvector** | PostgreSQL 扩展 | 与 Postgres 集成，SQL 查询 | 已有 Postgres 基础设施 |
| **Redis** | 自托管 / 托管 | 熟悉度高，支持向量+传统数据 | 已有 Redis 基础设施 |

### 核心选型维度

1. **规模**：百万级以下几乎所有数据库都能胜任；十亿级以上优先考虑 Milvus。
2. **混合搜索需求**：是否需要同时做向量搜索和关键词搜索？Weaviate、Qdrant、pgvector 支持较好。
3. **元数据过滤**：检索时是否需要复杂的过滤条件？Qdrant 和 Milvus 的过滤能力最强。
4. **运维成本**：Pinecone 和 Zilliz 是托管服务，运维成本最低；Milvus 自托管最复杂。
5. **成本**：托管服务按用量计费，适合波动大的场景；自托管适合负载稳定、长期运行的场景。

### pgvector 快速入门

如果你已经有 PostgreSQL，pgvector 是最简单的选择。

```sql
-- 安装扩展
CREATE EXTENSION vector;

-- 创建表
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536),
    metadata JSONB
);

-- 创建 HNSW 索引
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);

-- 插入数据
INSERT INTO documents (content, embedding, metadata)
VALUES ('文档内容', '[0.1, 0.2, ...]', '{"source": "doc1.pdf"}');

-- 相似度搜索
SELECT content, 1 - (embedding <=> query_embedding) AS similarity
FROM documents
ORDER BY embedding <=> query_embedding
LIMIT 5;
```

### Qdrant 快速入门

Qdrant 是开源向量数据库中性能和功能平衡较好的选择。

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct

client = QdrantClient(url="http://localhost:6333")

# 创建集合
client.create_collection(
    collection_name="my_docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE)
)

# 插入数据
client.upsert(
    collection_name="my_docs",
    points=[
        PointStruct(
            id=1,
            vector=[0.1, 0.2, ...],
            payload={"content": "文档内容", "source": "doc1.pdf"}
        )
    ]
)

# 搜索
results = client.search(
    collection_name="my_docs",
    query_vector=[0.3, 0.1, ...],
    limit=5,
    filter={"must": [{"key": "source", "match": {"value": "doc1.pdf"}}]}
)
```

## 近似最近邻算法

在高维空间中进行精确的最近邻搜索（计算查询向量与所有文档向量的距离）计算量太大。近似最近邻（ANN）算法通过牺牲极小的精度来换取数量级的速度提升。

### HNSW（Hierarchical Navigable Small World）

HNSW 是目前最流行的 ANN 算法，被 Pinecone、Qdrant、pgvector 等数据库广泛采用。

**核心思想**：构建多层图结构，上层是稀疏的"高速公路"，下层是密集的"街道"。搜索时先在上层快速定位到大致区域，然后在下层精确搜索。

```
Layer 2 (稀疏):    A ───────────────→ C
                    ↑                  ↓
Layer 1 (中等):    A → B → D → E → C
                    ↑   ↑   ↑   ↑   ↑
Layer 0 (密集):    所有节点，包含全部连接
```

**优点**：

- 搜索速度快（微秒到毫秒级）。
- 支持增量插入，无需重建整个索引。
- 精度高，召回率通常在 95% 以上。

**关键参数**：

- `M`：每个节点的最大连接数。越大，图越密集，搜索精度越高，但内存和构建时间增加。常见值 16-64。
- `ef_construction`：构建索引时的搜索深度。越大，索引质量越高，构建越慢。
- `ef_search`：检索时的搜索深度。越大，精度越高，速度越慢。可在查询时动态调整。

### IVF（Inverted File Index）

IVF 先将向量空间划分为 K 个簇（通过 K-means 聚类），每个簇有一个中心点。检索时，先找到最近的几个簇中心，然后只在这些簇内搜索。

```
向量空间:
┌─────────────────────────────┐
│  簇 A    │  簇 B    │ 簇 C  │
│   · ·    │   ·  ·   │  · ·  │
│   ·*·    │   ·  ·   │  · ·  │  ← * 是查询向量
│   · ·    │   ·  ·   │  · ·  │
└─────────────────────────────┘

查询向量离簇 A 中心最近，只在簇 A 内搜索
```

**优点**：内存占用小，适合超大规模数据。
**缺点**：增量更新困难，新增数据可能需要重新聚类。

### 算法选择建议

| 场景 | 推荐算法 | 原因 |
|------|---------|------|
| 百万级以下，频繁更新 | HNSW | 增量友好，精度高 |
| 十亿级以上，静态数据 | IVF + PQ | 内存效率优先 |
| 内存极度受限 | PQ / SQ | 向量压缩，牺牲精度换空间 |
| 需要精确结果（Top-1） | HNSW 提高 ef_search | HNSW 精度上限更高 |

## 索引构建的最佳实践

### 1. 批量插入

向量数据库的批量插入通常比逐条插入快一个数量级。

```python
# 错误：逐条插入
for doc in documents:
    client.upsert(collection, [doc])  # 慢！

# 正确：批量插入
batch_size = 1000
for i in range(0, len(documents), batch_size):
    batch = documents[i:i+batch_size]
    client.upsert(collection, batch)  # 快！
```

### 2. 分区与分片

对于大规模数据，考虑按元数据分区存储。例如，按时间分区（每月一个集合），或按类别分区。

```python
# 按年份分区
collections = ["docs_2024", "docs_2025", "docs_2026"]

# 检索时先确定目标分区
target = f"docs_{query_year}"
results = client.search(target, query_vector, limit=5)
```

### 3. 监控索引质量

定期评估索引的召回率（recall@k）：用已知的查询-答案对，检查正确答案是否出现在 Top-K 结果中。

```python
def evaluate_recall(test_cases, k=5):
    correct = 0
    for query, expected_doc_id in test_cases:
        results = vector_search(query, top_k=k)
        if expected_doc_id in [r.id for r in results]:
            correct += 1
    return correct / len(test_cases)

# 召回率低于阈值时，考虑调整索引参数或重建索引
```

## 小结

- 嵌入模型将文本映射到向量空间，是 RAG 的核心。选择时需考虑维度、语言支持、最大长度。
- 双编码器架构追求速度，适合大规模检索；交叉编码器追求精度，适合重排序。
- 向量数据库选型要考虑规模、混合搜索需求、运维成本和过滤能力。
- HNSW 是目前最主流的 ANN 算法，精度高且支持增量更新。
- 索引构建时注意批量插入、合理分区和定期质量监控。
