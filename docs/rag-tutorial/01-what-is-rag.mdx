# 什么是 RAG？

大语言模型（LLM）已经展现出惊人的语言能力，但它们有一个根本性的限制：**知识是静态的、有限的**。模型在训练时学到的知识被冻结在参数中，无法访问训练数据之外的信息，也无法感知实时变化。这导致了三个核心问题：

- **幻觉（Hallucination）**：模型在缺乏知识时，会自信地编造看似合理但完全错误的内容。
- **知识过时**：模型不知道训练截止日期之后发生的事件、发布的论文或更新的文档。
- **领域盲区**：对于企业内部文档、私有数据或高度专业化的领域，模型完全没有接触过。

RAG（Retrieval-Augmented Generation，检索增强生成）正是为了解决这些问题而诞生的架构范式。

## 核心思想

RAG 的基本思路非常直观：与其让模型凭空回答，不如先让它去查资料，然后基于查到的资料来回答。

> RAG 是一种将外部知识检索与大语言模型生成能力相结合的架构。在生成回答之前，系统先从外部知识库中检索相关信息，将这些信息作为上下文提供给模型，从而生成更准确、更可信、更有时效性的回答。

用一句话概括：**先检索，再生成**。

```
用户提问 → 检索相关文档 → 将文档+问题一起输入模型 → 模型生成回答
```

这种方式把模型从"闭卷考试"变成了"开卷考试"——它仍然需要理解和推理能力，但不再需要死记硬背所有知识。

<Collapse question="为什么 RAG 比微调更适合知识频繁更新的场景？">

微调的代价在于**需要重新训练模型**。每次知识更新，你都需要准备训练数据、启动训练任务、验证模型效果、部署新版本——这是一个完整的机器学习工程 pipeline。而 RAG 只需要更新知识库中的文档，系统立即就能检索到最新内容。从成本角度看，微调需要算力（GPU 小时）和标注数据，RAG 只需要向量数据库的存储和检索开销。从时效性看，微调一个模型可能需要数小时到数天，而更新知识库是毫秒级的。

另一个关键差异是**可解释性**。RAG 可以明确展示「我基于这几篇文档回答你」，用户可以自行验证来源的可靠性；而微调后的模型行为变化是隐式的，你无法直接追溯它为什么给出某个答案。

当然，RAG 和微调不是二选一的关系。在需要模型掌握特定推理模式（如某种代码风格、特定的逻辑链）时，微调仍然是必要的。
</Collapse>

<Collapse question="RAG 的检索阶段会引入哪些新的挑战？">

检索质量直接决定了 RAG 系统的上限。以下是几个核心挑战：

- **语义鸿沟**：用户的查询和文档的表述方式可能完全不同。例如用户问「怎么让 LLM 不瞎编」，而文档中写的是「缓解大语言模型幻觉的技术方案」。检索系统需要理解语义等价性，而不仅是关键词匹配。
- **文档切分粒度**：如果 chunk 太大，会包含大量无关信息，稀释上下文；如果 chunk 太小，可能丢失必要的上下文（如「上述方法」指代的内容）。
- **多跳检索**：有些问题需要结合多个文档才能回答。例如「比较 A 公司和 B 公司的 Q3 营收」——需要先分别检索 A 和 B 的财报，再综合对比。
- **负样本污染**：知识库中可能存在矛盾或过时的信息，检索阶段需要有能力区分可信度。

这些问题会在后续章节中详细展开解决方案。
</Collapse>

## RAG vs 微调

很多人会在 RAG 和微调（Fine-tuning）之间犹豫。两者不是互斥的，而是解决不同问题的工具。

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 即时生效，只需更新知识库 | 需要重新训练，成本高 |
| 幻觉控制 | 显著降低，答案可追溯 | 有限改善，模型仍可能编造 |
| 实现成本 | 低，无需训练 | 高，需要算力和数据 |
| 适用场景 | 知识频繁变化、需要引用来源 | 需要改变模型行为风格或提升特定任务能力 |
| 可解释性 | 高，可以展示检索到的来源 | 低，难以解释模型为何这样回答 |
| 数据量要求 | 少量文档即可工作 | 需要大量高质量标注数据 |

<Callout type="info" title="最佳实践">
  在大多数生产场景中，RAG 是第一选择。只有在 RAG 无法满足需求（如需要模型掌握某种特定的推理风格或输出格式）时，才考虑结合微调。
</Callout>

## RAG 能解决什么问题

### 1. 问答系统
企业内部的文档问答是 RAG 最典型的应用。员工可以向系统提问关于产品规格、流程规范、技术文档的问题，系统从企业知识库中检索答案。

### 2. 客服机器人
结合产品文档、FAQ、历史工单，RAG 驱动的客服机器人可以回答高度具体的客户问题，而不需要为每个问题硬编码回复。

### 3. 代码辅助
将代码库、API 文档、技术博客作为知识源，RAG 可以帮助开发者理解不熟悉的代码、查找相关实现示例。

### 4. 研究报告生成
从大量论文、报告、新闻中检索相关信息，然后综合生成研究报告或摘要。

### 5. 法律与医疗咨询
在法律和医疗领域，准确性至关重要。RAG 可以确保回答基于最新的法规条文或医学文献，并且可追溯来源。

## RAG 的局限性

RAG 不是万能的。理解它的边界同样重要：

- **检索质量决定上限**：如果检索阶段没有找到正确的文档，生成阶段再好也没用。Garbage in, garbage out。
- **上下文长度限制**：模型有 token 上限，无法一次性塞入太多检索结果。如何在有限的窗口内放入最有价值的信息是个挑战。
- **复杂推理仍有限**：对于需要多步推理、跨文档关联的问题，基础 RAG 可能力不从心。
- **多模态支持**：传统 RAG 主要处理文本，处理图像、表格、音频等多模态内容需要额外的工作。

<Callout type="warning" title="现实预期">
  RAG 能显著降低幻觉，但不能完全消除。如果检索到的文档本身包含错误信息，或者模型错误理解了文档内容，仍然会产生不准确的回答。始终建议在关键场景保留人工审核环节。
</Callout>

## 一个简单的 RAG 伪代码

下面是一个最基础的 RAG 系统的骨架，帮助你建立整体印象。后续章节会逐一展开每个环节：

```python
class SimpleRAG:
    def __init__(self, knowledge_base, embedding_model, llm):
        self.knowledge_base = knowledge_base
        self.embedding_model = embedding_model
        self.llm = llm

    def answer(self, query: str) -> str:
        # 1. 将查询转换为向量
        query_embedding = self.embedding_model.encode(query)

        # 2. 检索最相关的文档片段
        relevant_docs = self.knowledge_base.search(query_embedding, top_k=5)

        # 3. 构建提示，将检索结果作为上下文
        context = "\n\n".join([doc.text for doc in relevant_docs])
        prompt = f"""基于以下参考资料回答问题：

参考资料：
{context}

问题：{query}
"""

        # 4. 调用大模型生成回答
        return self.llm.generate(prompt)
```

这个流程看似简单，但每个环节都有大量的工程细节和优化空间。从下一章开始，我们将深入拆解完整的 RAG 流程。

## 小结

- RAG 的核心思想是"先检索，再生成"，让模型基于外部知识回答。
- RAG 相比微调，成本更低、更新更快、可解释性更强。
- RAG 适用于知识频繁变化、需要引用来源的场景。
- RAG 的效果上限由检索质量决定，不是零成本的解决方案。
