# 高级技术

基础 RAG（先检索再生成）已经能够解决很多问题，但在面对复杂查询、多跳推理、动态知识等挑战时，需要更先进的架构。本章介绍 RAG 的进阶形态。

## Agentic RAG

Agentic RAG 将 RAG 与 Agent 架构结合，让系统能够自主决策何时检索、检索什么、如何改进检索。

### 为什么需要 Agentic RAG

基础 RAG 的问题是流程固定的：

1. 用户提问
2. 检索相关文档
3. 生成回答

这个流程对简单问题有效，但对以下场景力不从心：

- **多跳问题**：需要跨多个文档、多次检索才能回答。例如"比较 A 公司和 B 公司在 2025 年的营收，哪个增长更快？"
- **检索失败**：第一次检索没有找到相关信息，需要改写查询重新检索。
- **工具组合**：除了检索知识库，还需要调用计算器、API、数据库等外部工具。
- **自我验证**：生成回答后，需要验证其正确性，如果不正确则重新检索。

### Agentic RAG 的架构

```
用户提问
    ↓
┌─────────────────────────┐
│  Agent 决策：是否需要检索？ │
│  - 是 → 执行检索         │
│  - 否 → 直接回答         │
└─────────────────────────┘
    ↓
检索结果评估：信息是否足够？
    ├── 是 → 生成回答
    │         ↓
    │      回答质量评估
    │         ├── 满意 → 返回用户
    │         └── 不满意 → 重新检索（循环）
    └── 否 → 改写查询 → 重新检索（循环）
```

### ReAct 模式在 RAG 中的应用

ReAct（Reasoning + Acting）是最常用的 Agentic RAG 模式。

```python
class AgenticRAG:
    def __init__(self, retriever, llm):
        self.retriever = retriever
        self.llm = llm

    def run(self, query: str, max_steps: int = 5) -> str:
        thoughts = []
        actions = []
        observations = []

        for step in range(max_steps):
            # 构建 ReAct 提示
            prompt = self.build_react_prompt(
                query=query,
                thoughts=thoughts,
                actions=actions,
                observations=observations
            )

            # LLM 决定下一步行动
            response = self.llm.generate(prompt)
            action = self.parse_action(response)

            if action.type == "answer":
                return action.content

            elif action.type == "retrieve":
                docs = self.retriever.search(action.query)
                observations.append(f"检索结果：{summarize_docs(docs)}")
                actions.append(action)

            elif action.type == "calculate":
                result = eval(action.expression)
                observations.append(f"计算结果：{result}")
                actions.append(action)

            thoughts.append(response)

        return "无法在规定时间内找到答案"
```

**ReAct 提示示例**：

```
你可以使用以下工具：
- retrieve(query): 从知识库检索相关信息
- calculate(expression): 执行数学计算

请按以下格式思考：
Thought: 你的推理过程
Action: 工具名称（输入参数）
Observation: 工具返回的结果

问题：2024年A公司营收100亿，增长20%；B公司营收150亿，增长15%。哪家公司增长额更高？

Thought: 我需要先计算两家公司的增长额，然后比较。
Action: calculate(100 * 0.20)
Observation: 20
Thought: A公司增长额是20亿。接下来计算B公司的增长额。
Action: calculate(150 * 0.15)
Observation: 22.5
Thought: B公司增长额是22.5亿，高于A公司的20亿。我可以回答问题了。
Action: answer(B公司增长额更高，为22.5亿元，而A公司为20亿元。)
```

### Self-RAG

Self-RAG 让模型在生成过程中自主判断是否需要检索，并评估检索结果和生成内容的质量。

```python
class SelfRAG:
    def generate(self, query: str) -> str:
        tokens = []

        while not self.is_complete(tokens):
            # 生成下一个 token，同时预测是否需要检索
            next_token, need_retrieve = self.llm.generate_with_retrieval_signal(
                query, tokens
            )

            if need_retrieve:
                # 用当前上下文生成检索查询
                retrieval_query = self.generate_retrieval_query(query, tokens)
                docs = self.retriever.search(retrieval_query)

                # 评估每个检索结果的相关性
                for doc in docs:
                    is_relevant = self.llm.judge_relevance(tokens, doc)
                    if is_relevant:
                        tokens.extend(["[引用]", doc.content])
                        break

            tokens.append(next_token)

        return "".join(tokens)
```

Self-RAG 的创新点在于：不是一次性检索所有内容，而是在生成过程中**按需检索**——模型自己判断什么时候需要查资料、查什么资料。

## Graph RAG

Graph RAG 利用知识图谱来增强检索，特别适合需要理解实体关系和跨文档推理的场景。

### 基础 RAG 的局限

基础 RAG 将文档切分为独立的片段，丢失了文档之间的关联信息：

- 人物关系："张三"和"李四"是什么关系？
- 因果关系：事件 A 导致了事件 B，但 A 和 B 在不同文档中。
- 层级关系：组织架构、产品分类等结构化信息。

### Graph RAG 架构

```
原始文档
    ↓
实体提取（人、组织、地点、概念）
    ↓
关系抽取（A 是 B 的 CEO，C 属于 D 部门）
    ↓
构建知识图谱（节点 = 实体，边 = 关系）
    ↓
社区检测（识别紧密相关的实体群组）
    ↓
为每个社区生成摘要

查询时：
用户问题 → 识别涉及实体 → 图谱检索相关子图 → 结合社区摘要生成回答
```

### Graph RAG 实现流程

```python
class GraphRAG:
    def build_graph(self, documents):
        """从文档构建知识图谱"""
        graph = nx.Graph()

        for doc in documents:
            # 1. 提取实体
            entities = self.extract_entities(doc.content)

            # 2. 提取关系
            relations = self.extract_relations(doc.content, entities)

            # 3. 添加到图谱
            for entity in entities:
                graph.add_node(entity.name, type=entity.type)

            for rel in relations:
                graph.add_edge(rel.source, rel.target, relation=rel.type)

        # 4. 社区检测
        communities = nx.community.louvain_communities(graph)

        # 5. 为每个社区生成摘要
        self.community_summaries = {}
        for i, community in enumerate(communities):
            subgraph = graph.subgraph(community)
            summary = self.generate_community_summary(subgraph)
            self.community_summaries[i] = summary

        self.graph = graph

    def answer(self, query: str) -> str:
        """基于图谱回答查询"""
        # 1. 从查询中提取实体
        query_entities = self.extract_entities(query)

        # 2. 在图谱中查找相关子图
        relevant_nodes = []
        for entity in query_entities:
            if entity.name in self.graph:
                # 获取邻居节点（一度关系）
                neighbors = list(self.graph.neighbors(entity.name))
                relevant_nodes.extend([entity.name] + neighbors)

        # 3. 获取相关社区的摘要
        relevant_communities = self.find_communities(relevant_nodes)
        context = "\n".join([
            self.community_summaries[c] for c in relevant_communities
        ])

        # 4. 生成回答
        return self.llm.generate(f"基于以下信息回答问题：\n{context}\n\n问题：{query}")
```

### Graph RAG vs 基础 RAG

| 维度 | 基础 RAG | Graph RAG |
|------|---------|-----------|
| 擅长的问题 | 事实查找、定义解释 | 关系推理、多跳查询 |
| 构建成本 | 低 | 高（需要实体和关系抽取） |
| 维护成本 | 低 | 中（图谱需要随文档更新） |
| 可解释性 | 中（可追溯到文档） | 高（可展示关系路径） |
| 适用数据 | 通用文档 | 人物、组织、事件密集的文档 |

<Callout type="info" title="何时使用 Graph RAG">
  Graph RAG 的构建和维护成本较高。建议仅在以下场景考虑：
  1. 用户频繁询问实体之间的关系（"谁是谁的上司？"）
  2. 文档之间存在大量隐式关联，向量相似度无法捕捉
  3. 需要可追溯的推理路径（如法律分析、尽职调查）
</Callout>

## 多模态 RAG

传统 RAG 处理的是纯文本，但真实世界的知识以多种形式存在：图片、表格、音频、视频。

### 多模态 RAG 架构

```
文本文档 ──→ 文本嵌入模型 ──→ 文本向量
图片     ──→ 视觉编码器（如 CLIP）──→ 图像向量
表格     ──→ 结构化文本化 ──→ 文本嵌入 ──→ 向量
音频     ──→ Whisper 转录 ──→ 文本嵌入 ──→ 向量
```

### 统一向量空间

使用多模态嵌入模型（如 CLIP、Jina-CLIP）将不同模态映射到同一向量空间：

```python
from transformers import CLIPModel, CLIPProcessor

model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

# 编码图片
image_inputs = processor(images=image, return_tensors="pt")
image_vector = model.get_image_features(**image_inputs)

# 编码文本
text_inputs = processor(text="一只猫在沙发上", return_tensors="pt")
text_vector = model.get_text_features(**text_inputs)

# 两者在同一空间中，可以直接计算相似度
similarity = cosine_similarity(image_vector, text_vector)
```

### 表格处理

表格是文档中常见但难以处理的多模态内容。

```python
def table_to_text(table_data: list[list]) -> str:
    """将表格转换为结构化文本描述"""
    headers = table_data[0]
    rows = table_data[1:]

    descriptions = []
    for row in rows:
        row_desc = "；".join([
            f"{headers[i]}为{row[i]}"
            for i in range(len(headers))
        ])
        descriptions.append(row_desc)

    return "\n".join(descriptions)

# 示例
table = [
    ["产品", "价格", "库存"],
    ["iPhone 15", "5999", "100"],
    ["iPad Pro", "8999", "50"]
]

print(table_to_text(table))
# 产品为iPhone 15；价格为5999；库存为100
# 产品为iPad Pro；价格为8999；库存为50
```

更高级的方案：使用专门的表格理解模型（如 Table-Transformer）提取表格结构，然后让 LLM 生成表格的自然语言摘要。

## 自适应 RAG

自适应 RAG 根据查询的复杂度，动态选择不同的检索和生成策略。

### 查询复杂度分类

```python
class AdaptiveRAG:
    def route_query(self, query: str) -> str:
        """判断查询复杂度，选择处理策略"""

        prompt = f"""请将以下查询分类为以下类型之一：
- simple: 简单事实查询，可以直接回答
- complex: 复杂查询，需要多步推理或多文档综合
- multi_hop: 多跳查询，需要跨文档关联信息

查询：{query}

类型："""

        query_type = self.llm.generate(prompt).strip().lower()

        if query_type == "simple":
            return self.no_retrieval_answer(query)
        elif query_type == "complex":
            return self.single_retrieval_answer(query)
        else:
            return self.multi_hop_answer(query)

    def no_retrieval_answer(self, query: str) -> str:
        """不需要检索，直接用 LLM 回答"""
        return self.llm.generate(query)

    def single_retrieval_answer(self, query: str) -> str:
        """基础 RAG：单次检索 + 生成"""
        docs = self.retriever.search(query)
        return self.generate_with_context(query, docs)

    def multi_hop_answer(self, query: str) -> str:
        """多跳 RAG：迭代检索 + 推理"""
        return self.agentic_rag.run(query)
```

### 路由决策的依据

除了让 LLM 分类，还可以用更轻量的方法：

```python
def classify_query_simple(query: str) -> str:
    """基于规则快速分类查询复杂度"""

    # 包含比较词 → 复杂
    comparison_words = ['比较', '对比', '差异', '区别', 'vs', 'versus']
    if any(w in query for w in comparison_words):
        return 'complex'

    # 包含多个实体 → 可能是多跳
    entities = extract_entities(query)
    if len(entities) >= 3:
        return 'multi_hop'

    # 包含"为什么""如何" → 复杂
    if any(w in query for w in ['为什么', '如何', '怎么', '原因']):
        return 'complex'

    # 简短的事实查询 → 简单
    if len(query) < 20 and '什么' in query:
        return 'simple'

    return 'complex'  # 默认保守策略
```

## RAG 与微调的结合

RAG 和微调不是互斥的，它们解决不同层面的问题。

### 何时结合使用

| 问题 | 解决方案 |
|------|---------|
| 模型不擅长遵循 RAG 提示格式 | 微调模型的指令遵循能力 |
| 领域术语理解不准确 | 微调嵌入模型 + RAG |
| 回答风格不符合要求 | 微调生成模型的风格 |
| 模型经常在 RAG 中幻觉 | 微调模型的忠实度（faithfulness） |

### 嵌入模型微调

对嵌入模型进行领域微调，使其更擅长理解你的特定领域术语：

```python
from sentence_transformers import SentenceTransformer, InputExample, losses

model = SentenceTransformer('BAAI/bge-m3')

# 准备训练数据：查询-文档正例对
train_examples = [
    InputExample(texts=["查询1", "相关文档1"]),
    InputExample(texts=["查询2", "相关文档2"]),
    # ...
]

train_dataloader = DataLoader(train_examples, shuffle=True, batch_size=16)
train_loss = losses.MultipleNegativesRankingLoss(model)

model.fit(train_objectives=[(train_dataloader, train_loss)], epochs=3)
```

### 生成模型微调

通过监督微调（SFT）让模型更好地遵循 RAG 的提示格式和约束：

```python
# 训练数据格式
sft_data = [
    {
        "messages": [
            {"role": "system", "content": "你是一个基于参考资料回答问题的助手..."},
            {"role": "user", "content": "参考资料：...\n问题：..."},
            {"role": "assistant", "content": "根据 [1] ..."}
        ]
    }
]
```

<Callout type="warning" title="先 RAG 后微调">
  建议先搭建基础 RAG 系统并评估效果，只有当基础 RAG 的效果不达标时，才考虑微调。微调需要大量标注数据和算力，是更重的投入。
</Callout>

## 小结

- **Agentic RAG**：将检索决策权交给 Agent，支持多步推理、工具调用和自我纠正。
- **Self-RAG**：在生成过程中按需检索，模型自主判断何时需要外部信息。
- **Graph RAG**：利用知识图谱捕捉实体关系，擅长多跳推理和关系查询。
- **多模态 RAG**：扩展 RAG 到图片、表格、音频等非文本内容。
- **自适应 RAG**：根据查询复杂度动态选择处理策略，平衡成本和效果。
- RAG 和微调可以结合使用：RAG 解决知识更新问题，微调解决模型能力问题。
