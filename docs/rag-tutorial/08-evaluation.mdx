# 评估与优化

构建 RAG 系统只是第一步，持续的评估和优化才能让它真正可用。本章建立一套完整的 RAG 评估体系，并讲解基于评估结果的优化方法。

## 为什么评估很重要

RAG 系统由多个组件组成（分块、嵌入、检索、生成），任何一个环节的退化都会导致整体效果下降。没有评估，你就不知道：

- 检索是否找到了正确的文档？
- 生成的回答是否基于检索到的文档？
- 系统上线后效果是否在持续退化？
- 某次改动（换模型、改分块策略）是提升还是下降了？

## 评估数据集构建

评估的前提是有一组"黄金标准"——已知正确答案的查询-文档对。

### 数据集格式

```json
[
  {
    "query": "什么是 RAG？",
    "relevant_doc_ids": ["doc_001", "doc_003"],
    "expected_answer": "RAG（检索增强生成）是一种将外部知识检索与大语言模型生成能力相结合的架构...",
    "category": "概念解释"
  },
  {
    "query": "如何在 Python 中实现上下文管理器？",
    "relevant_doc_ids": ["doc_042"],
    "expected_answer": "可以通过定义 __enter__ 和 __exit__ 方法，或使用 @contextmanager 装饰器...",
    "category": "代码实现"
  }
]
```

### 构建方法

1. **人工标注**：由领域专家编写查询和标注相关文档。最准确但成本高。
2. **LLM 辅助生成**：用 LLM 基于文档生成可能的查询。

```python
def generate_synthetic_queries(documents, llm, num_queries_per_doc=3):
    """用 LLM 基于文档生成合成查询"""
    queries = []

    for doc in documents:
        prompt = f"""基于以下文档内容，生成 {num_queries_per_doc} 个用户可能会问的问题。
要求：
- 问题应该能够在文档中找到答案
- 问题类型多样化（what、how、why、compare）
- 用中文提问

文档内容：
{doc.content[:2000]}

问题列表："""

        response = llm.generate(prompt)
        doc_queries = [q.strip('- ') for q in response.split('\n') if q.strip()]

        for q in doc_queries:
            queries.append({
                "query": q,
                "relevant_doc_ids": [doc.id],
                "category": infer_category(q)
            })

    return queries
```

3. **真实日志**：收集线上用户的真实查询和反馈（点赞/点踩）。

<Callout type="info" title="数据量建议">
  评估数据集不需要很大，但要覆盖不同类型的查询。建议至少 50-100 条，覆盖概念解释、事实查询、比较类、操作步骤等常见类型。
</Callout>

## 检索质量评估

检索质量的评估独立于生成，专注于"是否找到了正确的文档"。

### 核心指标

#### 1. 召回率（Recall@K）

正确答案是否出现在检索结果的前 K 个中。

```python
def recall_at_k(retrieved_ids, relevant_ids, k=5):
    """计算 Recall@K"""
    retrieved_top_k = set(retrieved_ids[:k])
    relevant = set(relevant_ids)

    if not relevant:
        return 1.0  # 没有相关文档时默认满分

    return len(retrieved_top_k & relevant) / len(relevant)

# 示例
retrieved = ["doc_003", "doc_007", "doc_001", "doc_012", "doc_005"]
relevant = ["doc_001", "doc_003"]

print(recall_at_k(retrieved, relevant, k=5))  # 1.0（两个都找到了）
print(recall_at_k(retrieved, relevant, k=2))  # 0.5（只找到了 doc_003）
```

#### 2. 精确率（Precision@K）

前 K 个结果中有多少是相关的。

```python
def precision_at_k(retrieved_ids, relevant_ids, k=5):
    retrieved_top_k = set(retrieved_ids[:k])
    relevant = set(relevant_ids)

    if not retrieved_top_k:
        return 0.0

    return len(retrieved_top_k & relevant) / len(retrieved_top_k)
```

#### 3. MRR（Mean Reciprocal Rank）

第一个相关文档的排名倒数。如果第一个相关文档排在第 3 位，得分为 1/3。

```python
def mean_reciprocal_rank(retrieved_ids_list, relevant_ids_list):
    """计算 MRR"""
    scores = []
    for retrieved, relevant in zip(retrieved_ids_list, relevant_ids_list):
        for rank, doc_id in enumerate(retrieved, 1):
            if doc_id in relevant:
                scores.append(1.0 / rank)
                break
        else:
            scores.append(0.0)
    return sum(scores) / len(scores)
```

#### 4. NDCG（Normalized Discounted Cumulative Gain）

考虑排名位置的加权指标，排名越靠前权重越高。

```python
import numpy as np

def dcg_at_k(relevances, k=5):
    """计算 DCG"""
    relevances = np.array(relevances[:k])
    positions = np.arange(1, len(relevances) + 1)
    return np.sum(relevances / np.log2(positions + 1))

def ndcg_at_k(retrieved_ids, relevant_ids, k=5):
    """计算 NDCG@K"""
    relevances = [1.0 if doc_id in relevant_ids else 0.0 for doc_id in retrieved_ids[:k]]
    ideal_relevances = sorted(relevances, reverse=True)

    dcg = dcg_at_k(relevances, k)
    idcg = dcg_at_k(ideal_relevances, k)

    return dcg / idcg if idcg > 0 else 0.0
```

### 检索评估完整流程

```python
def evaluate_retrieval(test_cases, retriever, ks=[1, 3, 5, 10]):
    """评估检索质量"""
    results = {f"recall@{k}": [] for k in ks}
    results.update({f"precision@{k}": [] for k in ks})
    mrr_scores = []

    for case in test_cases:
        query = case["query"]
        relevant = case["relevant_doc_ids"]

        retrieved = retriever.search(query, top_k=max(ks))
        retrieved_ids = [r.id for r in retrieved]

        for k in ks:
            results[f"recall@{k}"].append(recall_at_k(retrieved_ids, relevant, k))
            results[f"precision@{k}"].append(precision_at_k(retrieved_ids, relevant, k))

        mrr_scores.append(calculate_mrr_single(retrieved_ids, relevant))

    # 汇总
    summary = {}
    for metric, values in results.items():
        summary[metric] = sum(values) / len(values)
    summary["MRR"] = sum(mrr_scores) / len(mrr_scores)

    return summary
```

### 检索质量基准

| 指标 | 优秀 | 可接受 | 需改进 |
|------|------|-------|-------|
| Recall@5 | > 0.85 | 0.70 - 0.85 | < 0.70 |
| Recall@10 | > 0.90 | 0.80 - 0.90 | < 0.80 |
| MRR | > 0.80 | 0.60 - 0.80 | < 0.60 |

## 生成质量评估

生成质量的评估更复杂，因为它涉及语义理解和主观判断。

### 基于参考答案的评估

如果有标准答案，可以用以下方法：

#### 1. 语义相似度

用嵌入模型计算生成回答与标准答案的语义相似度。

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('BAAI/bge-m3')

def semantic_similarity(prediction, reference):
    emb_pred = model.encode(prediction)
    emb_ref = model.encode(reference)
    return cosine_similarity(emb_pred, emb_ref)
```

#### 2. LLM 作为评估器

让 LLM 扮演评估者，根据标准答案判断生成回答的质量。

```python
def llm_evaluate(prediction, reference, query):
    prompt = f"""请评估以下回答相对于参考答案的质量。

问题：{query}

参考答案：{reference}

生成回答：{prediction}

请从以下维度评分（1-5分）：
1. 正确性：回答是否包含正确信息？
2. 完整性：回答是否覆盖了问题的所有方面？
3. 忠实度：回答是否只基于应有知识，没有编造？

请输出 JSON 格式：
{{
  "correctness": 4,
  "completeness": 3,
  "faithfulness": 5,
  "explanation": "解释..."
}}"""

    return parse_json(llm.generate(prompt))
```

### 无参考答案的评估

很多时候没有标准答案，需要评估回答本身的质量：

#### 1. 忠实度（Faithfulness）

回答是否忠于检索到的文档，没有引入外部信息或幻觉。

```python
def check_faithfulness(answer, retrieved_docs):
    """检查回答中的每个事实是否能在文档中找到支持"""
    # 将回答分解为事实陈述
    facts = extract_facts(answer)

    all_doc_text = " ".join([d.content for d in retrieved_docs])
    unsupported = []

    for fact in facts:
        # 检查事实是否被文档蕴含
        if not is_entailed(fact, all_doc_text):
            unsupported.append(fact)

    faithfulness_score = 1.0 - len(unsupported) / len(facts) if facts else 1.0
    return faithfulness_score, unsupported
```

#### 2. 回答相关性（Answer Relevance）

回答是否直接针对用户问题，没有跑题。

```python
def answer_relevance(query, answer):
    """计算回答与问题的相关性"""
    query_embedding = embedding_model.encode(query)
    answer_embedding = embedding_model.encode(answer)
    return cosine_similarity(query_embedding, answer_embedding)
```

#### 3. 上下文相关性（Context Relevance）

检索到的文档是否与用户问题相关（避免检索到无关文档作为上下文）。

```python
def context_relevance(query, retrieved_docs):
    """计算检索文档与问题的平均相关性"""
    query_embedding = embedding_model.encode(query)
    scores = []

    for doc in retrieved_docs:
        doc_embedding = embedding_model.encode(doc.content)
        scores.append(cosine_similarity(query_embedding, doc_embedding))

    return sum(scores) / len(scores) if scores else 0.0
```

### RAGAS 框架

RAGAS 是一个专门用于 RAG 评估的开源框架，提供了完整的评估指标。

```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_relevancy

# 准备数据
dataset = Dataset.from_dict({
    "question": ["什么是 RAG？", "如何分块？"],
    "answer": ["RAG 是...", "分块的方法有..."],
    "contexts": [["doc1", "doc2"], ["doc3"]],
    "ground_truth": ["检索增强生成是...", "分块策略包括..."]
})

# 运行评估
results = evaluate(
    dataset=dataset,
    metrics=[faithfulness, answer_relevancy, context_relevancy]
)

print(results)
# {'faithfulness': 0.85, 'answer_relevancy': 0.92, 'context_relevancy': 0.78}
```

## 端到端评估

除了分别评估检索和生成，还需要评估整个系统的用户体验。

### A/B 测试

在线上环境运行两个版本的 RAG 系统，比较用户满意度。

```python
# 随机分配用户到不同版本
version = "A" if random.random() < 0.5 else "B"

if version == "A":
    answer = rag_v1.answer(query)
else:
    answer = rag_v2.answer(query)

# 记录用户反馈
feedback = collect_user_feedback(query, answer, version)
```

### 用户反馈指标

- **点赞/点踩率**：用户对回答的直接反馈。
- **追问率**：用户是否对同一问题继续追问（可能意味着回答不完整）。
- **引用点击率**：用户是否点击查看引用来源（可能意味着用户对回答存疑）。
- **会话时长**：过长可能意味着用户在反复尝试获得满意答案。

## 基于评估的优化策略

评估的目的不是评分，而是指导优化。以下是对应不同问题的优化策略。

### 问题：检索召回率低

**诊断**：Recall@K 低，说明正确答案不在检索结果中。

**优化方向**：

| 可能原因 | 解决方案 |
|---------|---------|
| 分块过小，关键信息被切断 | 增大分块大小或使用父文档检索 |
| 查询与文档表达方式差异大 | 使用 HyDE 或多查询生成 |
| 嵌入模型不适合领域 | 换用领域优化的嵌入模型或微调 |
| 专有名词匹配失败 | 添加混合搜索（BM25） |
| 查询过短，语义不明确 | 查询扩展或改写 |

### 问题：检索精确率低

**诊断**：Precision@K 低，检索结果中混入了很多无关文档。

**优化方向**：

| 可能原因 | 解决方案 |
|---------|---------|
| 分块过大，包含无关内容 | 减小分块大小或优化分块边界 |
| 相似但无关的文档太多 | 添加重排序（Reranking） |
| 嵌入模型过于宽泛 | 换用更精确的嵌入模型 |
| 缺少元数据过滤 | 添加前置过滤缩小搜索范围 |

### 问题：回答幻觉严重

**诊断**：Faithfulness 低，回答中包含文档中没有的信息。

**优化方向**：

| 可能原因 | 解决方案 |
|---------|---------|
| 提示约束不够强 | 强化系统提示，明确"只基于资料回答" |
| 检索到的文档不相关 | 先解决检索质量问题 |
| 模型温度过高 | 降低 temperature 至 0.0-0.2 |
| 缺少拒绝回答机制 | 添加"无法回答"的判断逻辑 |
| 缺少自我验证 | 添加 Self-Check 步骤 |

### 问题：回答不完整

**诊断**：Completeness 低，回答遗漏了问题的某些方面。

**优化方向**：

| 可能原因 | 解决方案 |
|---------|---------|
| Top-K 太小，遗漏关键文档 | 增大 K 值或使用多查询 |
| 上下文窗口限制 | 上下文压缩或分层检索 |
| 提示没有要求全面回答 | 在提示中添加"请覆盖所有方面" |
| 问题包含多个子问题 | 使用查询分解，分别检索回答 |

## 持续监控

RAG 系统上线后，需要建立持续监控机制。

### 监控指标

```python
class RAGMonitor:
    def __init__(self):
        self.metrics = {
            "query_count": 0,
            "avg_retrieval_time": [],
            "avg_generation_time": [],
            "retrieval_recall": [],
            "user_feedback": [],
        }

    def log_query(self, query, retrieved_docs, answer, latency):
        self.metrics["query_count"] += 1
        self.metrics["avg_retrieval_time"].append(latency["retrieval"])
        self.metrics["avg_generation_time"].append(latency["generation"])

    def log_feedback(self, query, answer, feedback: bool):
        """记录用户反馈（True=满意，False=不满意）"""
        self.metrics["user_feedback"].append(feedback)

    def get_dashboard(self):
        return {
            "total_queries": self.metrics["query_count"],
            "avg_retrieval_ms": sum(self.metrics["avg_retrieval_time"]) / len(self.metrics["avg_retrieval_time"]),
            "avg_generation_ms": sum(self.metrics["avg_generation_time"]) / len(self.metrics["avg_generation_time"]),
            "user_satisfaction": sum(self.metrics["user_feedback"]) / len(self.metrics["user_feedback"]),
        }
```

### 告警规则

- 用户满意度连续下降超过 5% → 触发评估检查
- 平均检索延迟超过 500ms → 检查索引和基础设施
- 幻觉检测失败率超过 10% → 检查提示和模型配置

## 优化检查清单

在每次迭代优化前，用这个清单诊断问题：

**检索端**：
- [ ] 分块大小是否合理？（检查块长度分布）
- [ ] 嵌入模型是否与数据语言匹配？
- [ ] 是否需要混合搜索？（测试含专有名词的查询）
- [ ] 重排序是否提升了 Top-5 精度？
- [ ] 元数据过滤是否有效利用了？

**生成端**：
- [ ] 提示是否明确约束了回答边界？
- [ ] 上下文长度是否充分利用了模型窗口？
- [ ] temperature 是否足够低？
- [ ] 是否有"无法回答"的拒绝机制？
- [ ] 引用是否准确可追溯？

**系统端**：
- [ ] 端到端延迟是否可接受？
- [ ] 是否有监控和告警？
- [ ] 评估数据集是否覆盖了主要查询类型？
- [ ] 用户反馈收集机制是否完善？

## 小结

- 评估是 RAG 系统持续改进的基础，没有评估就无法判断优化方向。
- 检索评估关注 Recall@K、Precision@K、MRR 等指标，可以用客观数据衡量。
- 生成评估关注忠实度、相关性、完整性，可以结合 LLM 评估和人工判断。
- RAGAS 提供了开箱即用的 RAG 评估框架。
- 基于评估结果的优化要有针对性：先诊断问题所在环节，再选择对应策略。
- 上线后的持续监控和用户反馈收集与离线评估同等重要。
