# RAG 通用流程

一个完整的 RAG 系统可以拆解为五个核心阶段。理解这些阶段以及它们之间的数据流转，是构建和优化 RAG 的基础。

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   数据准备   │ → │   索引构建   │ → │   查询处理   │ → │   检索召回   │ → │   答案生成   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

这五个阶段分别是：**数据准备**、**索引构建**、**查询处理**、**检索召回**、**答案生成**。前两个阶段通常在线下完成（或异步执行），后三个阶段在线上响应用户查询时实时执行。

## 阶段一：数据准备

这是整个流程的起点。原始数据通常以各种格式存在——PDF、Word、网页、数据库记录、Markdown 文件等。数据准备阶段的目标是将这些异构数据转换为统一的、可供后续处理的文本格式。

```
原始文档（PDF/Word/HTML/数据库）
    ↓
内容提取（去除格式、保留语义）
    ↓
文本清洗（去除噪声、规范化）
    ↓
文档分块（Chunking）
    ↓
文本片段列表
```

**关键操作**：

- **格式解析**：PDF 需要处理表格、图片标题、页眉页脚；HTML 需要过滤导航栏、广告等无关内容。
- **文本清洗**：去除重复空格、统一编码、处理特殊字符。
- **文档分块**：将长文档切分成合适大小的片段。这是最关键的一步，直接影响检索精度。分块策略包括固定长度分块、按段落分块、按语义分块等。
- **元数据附加**：为每个片段附加来源信息（文件名、页码、章节标题、发布时间等），便于后续引用和过滤。

<Callout type="info" title="分块大小">
  分块大小没有银弹。太小的块（如 100 字）会丢失上下文；太大的块（如 2000 字）会稀释关键信息，且超出嵌入模型的处理限制。常见的起点是 300-500 token，配合 50-100 token 的重叠（overlap）。
</Callout>

## 阶段二：索引构建

索引构建是将文本片段转化为可快速检索的结构的过程。现代 RAG 系统几乎全部采用向量索引。

```
文本片段列表
    ↓
嵌入模型编码（Embedding）
    ↓
高维向量（如 768 维或 1536 维）
    ↓
向量数据库索引（HNSW / IVF 等）
    ↓
可检索的向量存储
```

**关键操作**：

- **向量化**：使用预训练的嵌入模型（如 OpenAI text-embedding-3、BGE、E5 等）将每个文本片段编码为稠密向量。
- **索引算法**：向量数据库使用近似最近邻（ANN）算法构建索引，常见的有 HNSW（Hierarchical Navigable Small World）和 IVF（Inverted File Index）。
- **混合索引（可选）**：除了向量索引，还可以构建传统的倒排索引（关键词索引），用于关键词搜索或混合搜索场景。
- **存储设计**：需要考虑向量、原始文本、元数据的存储方案。有些数据库（如 Pinecone、Weaviate、Milvus）将三者统一管理；有些则需要自行维护映射关系。

索引构建是一次性的工作，但当知识库更新时，需要增量更新索引。设计时要考虑：

- 如何检测文档变更（新增、删除、修改）
- 增量更新还是全量重建
- 更新期间的查询一致性

## 阶段三：查询处理

当用户提出问题时，系统需要对查询进行处理，使其更适合检索。

```
用户原始查询
    ↓
查询理解（意图识别、实体提取）
    ↓
查询改写 / 扩展（可选）
    ↓
向量化（使用与索引相同的嵌入模型）
    ↓
查询向量
```

**关键操作**：

- **查询向量化**：使用与索引构建阶段完全相同的嵌入模型，将查询文本编码为向量。模型不一致会导致语义空间不匹配，检索效果大打折扣。
- **查询改写**：用户的原始查询往往不够精确。可以通过以下方式改进：
  - **查询扩展**：添加同义词或相关术语。
  - **HyDE（Hypothetical Document Embeddings）**：让模型先根据问题生成一个假想的答案文档，然后用这个假想文档去检索。这适用于问题很短但答案很长的情况。
  - **多查询生成**：让模型从不同角度重写问题，分别检索，合并结果。
- **元数据过滤**：如果查询中包含时间、类别等约束条件，可以先过滤元数据，缩小检索范围。

<Callout type="warning" title="模型一致性">
  索引构建和查询处理必须使用同一个嵌入模型。即使两个模型都号称 768 维，如果架构不同，向量空间完全不同，检索会失效。切换嵌入模型时，必须重建整个索引。
</Callout>

## 阶段四：检索召回

检索阶段根据查询向量，从索引中召回最相关的文档片段。

```
查询向量 + 可选的过滤条件
    ↓
近似最近邻搜索（ANN）
    ↓
Top-K 候选片段
    ↓
重排序 / 精排（可选）
    ↓
最终候选片段列表
```

**关键操作**：

- **向量相似度搜索**：计算查询向量与索引中所有文档向量的相似度（通常用余弦相似度或点积），返回最相似的 K 个。
- **Top-K 选择**：K 的选择是召回率和精度的权衡。K 太小可能遗漏关键信息；K 太大会增加后续生成的负担。常见取值是 5-20。
- **混合搜索**：同时运行向量搜索和关键词搜索，然后用加权融合（如 RRF，Reciprocal Rank Fusion）合并结果。这对于包含专有名词、代码、ID 的查询特别有效。
- **重排序（Reranking）**：使用专门的交叉编码器（Cross-Encoder）对召回的候选片段进行精排。第一阶段的双编码器（Bi-Encoder）追求速度，交叉编码器追求精度。典型的流程是先用双编码器召回 100 个，再用交叉编码器精排选出 Top-5。
- **上下文压缩（可选）**：如果召回的片段太长，可以进一步压缩，只保留与查询最相关的句子或段落。

## 阶段五：答案生成

最后一步是将检索到的信息喂给大语言模型，让它生成最终答案。

```
最终候选片段 + 用户原始查询
    ↓
提示工程（Prompt Engineering）
    ↓
大语言模型（LLM）
    ↓
结构化回答（可选：引用来源）
```

**关键操作**：

- **提示构建**：这是影响生成质量最关键的环节。一个经典的 RAG 提示模板如下：

```
你是一个专业的助手。请严格基于以下参考资料回答用户的问题。
如果参考资料中没有足够的信息，请明确说明"根据现有资料无法回答"。
回答时请引用参考资料的编号。

参考资料：
[1] {片段1内容}
[2] {片段2内容}
[3] {片段3内容}

用户问题：{query}
```

- **引用标注**：让模型在回答中标注信息来源（如"根据[1]..."），增强可解释性和可信度。
- **无法回答处理**：明确指示模型在资料不足时承认不知道，而不是编造答案。这是控制幻觉的最后一道防线。
- **后处理**：对模型输出进行格式化处理、去除冗余、提取结构化数据等。

## 数据流转总览

下面是完整的数据流转图：

```
离线阶段（异步）
═══════════════════════════════════════════════════════
PDF/Word/HTML/数据库  ──→  数据提取  ──→  清洗  ──→  分块  ──→  文本片段
                                                              ↓
                                                   嵌入模型编码
                                                              ↓
                                                   向量数据库索引构建


在线阶段（实时）
═══════════════════════════════════════════════════════
用户查询  ──→  查询改写  ──→  嵌入编码  ──→  向量检索(Top-K)
                                                  ↓
                                         重排序(Rerank)
                                                  ↓
                                         提示构建 + 上下文注入
                                                  ↓
                                         大模型生成回答
                                                  ↓
                                         带引用的结构化输出
```

## 各阶段之间的关键决策点

构建 RAG 系统时，每个阶段都需要做出关键决策：

| 阶段 | 关键决策 | 影响 |
|------|---------|------|
| 数据准备 | 分块策略和大小 | 直接影响检索精度 |
| 索引构建 | 嵌入模型选择 | 决定语义理解能力和向量质量 |
| 查询处理 | 是否做查询改写 | 影响召回率 |
| 检索召回 | K 值和是否重排序 | 精度与延迟的权衡 |
| 答案生成 | 提示模板设计 | 决定回答质量和幻觉控制 |

## 小结

- RAG 流水线分为五个阶段：数据准备 → 索引构建 → 查询处理 → 检索召回 → 答案生成。
- 前两阶段离线执行，后三阶段在线实时响应。
- 每个阶段都有大量的工程选择和优化空间。
- 整个系统的瓶颈通常是检索质量：检索错了，生成再好也救不回来。
